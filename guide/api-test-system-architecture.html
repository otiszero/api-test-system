1. Tổng quan hệ thống

Mục đích: Framework tự động sinh và chạy API tests + E2E tests cho Upmount Custody Platform.

Tech stack:

  • Vitest Test runner (API tests)
  • Playwright Browser automation (E2E tests)
  • Axios HTTP client
  • AJV JSON Schema validation (OpenAPI)
  • Faker.js Realistic test data
  • Gherkin Feature files (Given/When/Then)
  • TypeScript Strict mode
Nguyên tắc cốt lõi: AI sinh test code MỘT LẦN (authoring-time) → tests chạy trực tiếp (zero runtime AI cost). Mọi endpoint phải có trong canonical-endpoints.json trước khi sinh test.

4. Test Pyramid (7 Layers)

LayerMục đíchConfig cầnVí dụ
01 Smoke Connectivity — mỗi endpoint trả response OpenAPI GET /health → 200
02 Contract Schema validation — response đúng format OpenAPI Response matches JSON Schema
03 Single Happy path + validation + errors per endpoint api + auth config POST /vault → 201, invalid body → 400
04 Integration Cross-resource workflows, state transitions test-rules.md Login → Profile → Create Vault → List Transactions
05 RBAC Permission enforcement, IDOR auth (2+ roles) User A cannot access User B's vault
06 Security SQL injection, XSS, auth bypass api + auth SQLi in query param → no data leak
07 DB DB state verification, constraints db.config.json DELETE vault → FK cascade check

5. Helpers — Hạ tầng test

api-client.ts

Axios wrapper. Mỗi request tự ghi evidence (method, URL, headers, response, duration, cURL). Token management qua setToken() / clearToken().

auth-helper.ts

Đọc auth.config.json. Hỗ trợ: bearer_direct (token sẵn), bearer (login endpoint), api_key, basic. Phân biệt role + label.

dependency-resolver.ts

Đọc dependency-graph.json. Cung cấp execution order, direct deps, transitive deps (mới), priority sorting.

cleanup.ts

Track created resources → cleanAll() xóa theo thứ tự ngược (LIFO). Tự set admin token khi cleanup.

test-data-factory.ts

Faker.js builders: register, login, KYB, vault, withdraw, file upload, org invite. Realistic data, không "test123".

schema-validator.ts

AJV + ajv-formats. Validate response body against OpenAPI schemas. Report mismatches chi tiết.

6. Anti-Hallucination System

Vấn đề

AI sinh test cho endpoints không tồn tại (vd: /markets/new-markets) → 404 failures.

Giải pháp: Canonical Endpoint Manifest
  1. scripts/extract-endpoints.ts parse OpenAPI spec programmatically
  2. Apply blacklist/whitelist filters từ api.config.json
  3. Output: generated/canonical-endpoints.json — SINGLE SOURCE OF TRUTH
  4. Mọi generate command PHẢI đọc file này trước
  5. CHỈ sinh test cho endpoints có isTestable: true
// canonical-endpoints.json
{
  "metadata": { "totalEndpoints": 76, "testableEndpoints": 42 },
  "endpoints": [
    { "method": "GET", "path": "/api/users/vault-accounts",
      "isTestable": true, "blacklistReason": null }
  ],
  "byResource": { "Vault Accounts": ["GET /api/users/vault-accounts", ...] }
}

3 cách tạo E2E test

CáchCommandInputOutputKhi nào dùng

/describe-e2e — Natural Language → Feature File

QC mô tả test bằng ngôn ngữ tự nhiên → AI sinh .feature file chuẩn Gherkin.

# QC nói:
/describe-e2e vault-management
"Tôi muốn test tạo vault mới, cần login trước,
 vào trang vault, bấm tạo, điền tên, nhập OTP,
 verify vault xuất hiện trong danh sách"

# AI sinh ra: e2e/features/vault-management.feature
@needs(login-owner)
Feature: Vault Management
  Scenario: Create new vault
    Given I am on the "vault" page
    When I click the "Create Vault" button
    And I fill "vault name" with "My New Vault"
    And I fill "OTP" with "123456"
    And I click the "Submit" button
    Then I should see "My New Vault"

Key features: Auto-detect login prerequisites → suggest @needs tags. Map natural language → catalog step patterns cho maximum match rate. Hỗ trợ tiếng Việt lẫn English.

8. withNeeds() — Integration Data Dependencies

Vấn đề trước đây
// Fragile pattern:
let vaultId: string | null = null;
beforeAll(() => { apiClient.setToken(token); });
it('test', () => {
  if (!vaultId) return;  // ← silent skip, unreliable
});
Giải pháp: withNeeds() wrapper
import { withNeeds } from '../../helpers/integration-needs';

// Chỉ khai báo leaf resource — transitive deps tự resolve
withNeeds(['Vault Accounts'], (ctx) => {
  // Auto-setup chain:
  //   Country → Users auth → User Profile → Organization Kyb → Vault Accounts
  describe('Vault tests', () => {
    it('list vaults', async () => {
      // ctx.ownerToken đã có sẵn
      // ctx.data.vaultId đã được populate
      const res = await apiClient.get('/api/users/vault-accounts');
      expect(res.status).toBe(200);
    });
  });
  // afterAll: cleanup.cleanAll() tự động
});

Cơ chế hoạt động

  1. withNeeds(['Vault Accounts']) gọi dependencyResolver.getTransitiveDependencies()
  2. Resolve đệ quy: Vault Accounts → deps: User Profile, Organization Kyb → deps: Users auth, Files, Country...
  3. Sort theo priority (từ dependency-graph.json)
  4. beforeAll: chạy từng provider theo thứ tự → populate ctx
  5. Tests dùng ctx.data.* — không cần let variables
  6. afterAll: cleanup.cleanAll() xóa theo thứ tự ngược

Available ctx fields

Users authctx.ownerToken, ctx.data.normalUserToken
Countryctx.data.countryId, ctx.data.countries
User Profilectx.data.profile, ctx.data.orgId
Filesctx.data.fileId, ctx.data.presignedPost
Organization Kybctx.data.kybStatus

11. Reports & Evidence

Mỗi API call qua apiClient tự ghi evidence:

  • Request: method, URL, headers, body, cURL command
  • Response: status, headers, body (truncated nếu >2KB)
  • Timing: duration (ms), timestamp

Output files:

  • reports/latest-report.md — Test summary
  • reports/coverage-matrix.md — Endpoint × layer heatmap
  • reports/bugs-found.md — Failures & bugs
  • reports/detailed-evidence.html — Interactive evidence viewer
  • reports/html/index.html — Vitest HTML dashboard
