⚙️ 2. Config config/e2e.config.json

Key config fields ▶
appUrl
Frontend URL (e.g. https://dev.upmount.sotatek.works)
basicAuth
HTTP Basic Auth for protected environments
pages
Friendly name → URL path map. Used by I am on the "{page}" page
accounts
Named accounts with email, password, totpSecret for 2FA login
browser
chromium | firefox | webkit
timeout
Test timeout in ms (default: 30000)
screenshots
on-failure | on | off

🔗 2b. Reusable Suites config/suites.config.json

Define reusable prerequisite steps (login, navigation, setup) that features can include via @needs(suite-name) tags. Steps are injected into test.beforeEach() at generation time.

Available suites ▶
Suite nameWhat it doesType
login-ownerLogin as owner with 2FASteps
login-owner-no2faLogin as owner without 2FASteps
navigate-dashboardNavigate to dashboard pageSteps
navigate-loginNavigate to login pageSteps
logged-in-dashboardLogin + navigate to dashboard (combined)Composite

Usage in .feature files

# Single suite — login before all scenarios
@smoke @needs(login-owner)
Feature: Dashboard Access
  Scenario: Dashboard visible
    Then I should see "Dashboard"

# Multiple suites — chain dependencies
@needs(login-owner, navigate-dashboard)
Feature: Dashboard Widgets
  Scenario: Widgets loaded
    Then I should see "Vault"

# Composite suite — shorthand for multiple suites
@needs(logged-in-dashboard)
Feature: Dashboard Settings
  Scenario: Settings link visible
    Then I should see "Settings"
How it works
The generator sees @needs(login-owner), looks up the suite in suites.config.json, and injects its steps into test.beforeEach(). Composite suites (like logged-in-dashboard) resolve nested dependencies automatically. No duplicate steps — deduplication is built in.

Adding custom suites

// In config/suites.config.json
{
  "suites": {
    "my-custom-suite": {
      "description": "What this suite does",
      "steps": [
        { "keyword": "Given", "text": "I am on the \"login\" page" },
        { "keyword": "When", "text": "I fill input \"Email\" with \"test@test.com\"" }
      ]
    },
    "my-composite-suite": {
      "description": "Combines other suites",
      "needs": ["login-owner", "navigate-dashboard"]
    }
  }
}

💬 3. Natural Language → .feature /describe-e2e

Don't know Gherkin syntax? Describe your tests in plain Vietnamese or English, and AI generates a valid .feature file optimized for the step catalog.

How to use /describe-e2e ▶ 
# Run the command with a feature name
/describe-e2e dashboard-access

Then describe what you want to test:

Vietnamese example

Kiểm tra trang dashboard sau khi đăng nhập:
- Đăng nhập bằng tài khoản owner có 2FA
- Sau khi login, kiểm tra thấy text "Dashboard"
- URL phải chứa /dashboard
- Kiểm tra có nút "Settings" hiển thị

English example

Test login validation:
- Go to login page
- Try with invalid email format → should see "Email is invalid"
- Try with wrong password → should see "Incorrect credentials"
- Try with valid credentials → should redirect to dashboard

Output

AI generates e2e/features/dashboard-access.feature:

@smoke @needs(login-owner)
Feature: Dashboard Access
  Verify dashboard page is accessible after login.

  Scenario: Dashboard visible after login
    Then I should see "Dashboard"
    And the URL should contain "/dashboard"

  Scenario: Settings button visible
    Then I should see "Settings"
Smart behaviors
  • Auto-detects login prerequisites → adds @needs tags
  • Uses catalog step patterns for maximum match rate (= lower cost)
  • Converts similar cases into Scenario Outline with Examples table
  • Uses page/account names from e2e.config.json

After generating the .feature file, continue with:

/generate-e2e dashboard-access    # → .spec.ts
/run-e2e dashboard-access --headed  # → run with browser visible

✍️ 4. Write .feature Files (Manual)

Create files in e2e/features/ using Gherkin syntax. Use step patterns from the catalog below.

Example: Login validation ▶ 
@smoke
Feature: Login Flow
  Users can log in with valid credentials and see validation errors.

  Background:
    Given I am on the "login" page

  Scenario: Login page is visible
    Then I should see "Welcome back,"

  Scenario Outline: Login validation - <case>
    When I fill input "Enter email address" with "<email>"
    And I fill input "Enter password" with "<password>"
    And I click button "Continue"
    Then I should see "<message>"

    Examples:
      | case                 | email           | password       | message                                        |
      | invalid email format | notanemail      | TestPassword1! | Email address is invalid                       |
      | wrong credentials    | bad@example.com | TestPassword1! | Your email or password is incorrect, try again.|
Example: Login with 2FA ▶ 
@smoke
Feature: Login with 2FA
  Users can log in with email, password and TOTP 2FA code.

  Scenario: Login with 2FA using account config
    Given I login as "owner" with 2FA
    Then I should see "Dashboard"
How 2FA works
The I login as "{user}" with 2FA step reads credentials + TOTP secret from e2e.config.json → accounts, generates a real-time OTP code, types it into the 6-digit input, and clicks Verify.
Example: Hybrid UI + API ▶ 
Feature: Create Order
  Scenario: Submit order and verify via API
    Given API: user "owner" is authenticated
    Given I am on the "orders" page
    When I fill input "Amount" with "100"
    And I click button "Submit"
    Then I should see "Order created"
    When API: GET "/orders" returns 200
    Then API: response should contain "id"
Example: Using @needs tag (suite injection) ▶ 
@smoke @needs(login-owner)
Feature: Dashboard Access
  Verify dashboard page is accessible after login.

  Scenario: Dashboard visible after login
    Then I should see "Dashboard"
    And the URL should contain "/dashboard"
How @needs works
The @needs(login-owner) tag tells the generator to inject the login-with-2FA steps into test.beforeEach(). No need to write login steps manually — they're pulled from suites.config.json.
Template (copy to start) ▶ 
@smoke
Feature: [Feature Name]
  [Brief description]

  Background:
    Given I am on the "[page]" page

  Scenario: [Happy path]
    When I fill input "[placeholder]" with "[value]"
    And I click button "[Button Text]"
    Then I should see "[expected text]"

  Scenario Outline: [Validation] - <case>
    When I fill input "[field]" with "<input>"
    And I click button "[Button]"
    Then I should see "<message>"

    Examples:
      | case          | input     | message        |
      | empty field   |           | Required       |
      | invalid input | bad-value | Invalid format |

🔄 6. Generate Tests

# Generate all features
/generate-e2e

# Generate single feature
/generate-e2e login
/generate-e2e login-2fa

# Or run directly
npx tsx e2e/generator/generator.ts
npx tsx e2e/generator/generator.ts login-2fa

Output: e2e/generated/{feature-name}.spec.ts

Important
Generated specs are in e2e/generated/ — do NOT edit them manually. Change the .feature file and regenerate instead.

▶️ 7. Run Tests

# Run all E2E tests
/run-e2e

# Run specific feature
/run-e2e login-2fa

# Run with browser visible (headed mode)
/run-e2e --headed

# Run with trace viewer (debug failed tests)
/run-e2e --trace

# Run with Playwright debug mode (step-by-step)
/run-e2e --debug

# Or run directly
npx playwright test --config=e2e/playwright.config.ts
npx playwright test --config=e2e/playwright.config.ts e2e/generated/login-2fa.spec.ts --headed

🐛 8. Debug Failed Tests

View trace for failed test ▶ 
# After a failed test, open trace viewer
npx playwright show-trace e2e/test-results/{test-folder}/trace.zip

# View HTML report
npx playwright show-report reports/e2e

Traces show every action, screenshot at each step, network requests, and console logs.

Error context files ▶

On failure, Playwright saves an error-context.md in the test results folder with a DOM snapshot showing exactly what was on the page.

cat e2e/test-results/{test-folder}/error-context.md

⚡ 9. Quick Reference

Common workflows ▶

Add a new test scenario (manual Gherkin)

  1. Create/edit e2e/features/my-feature.feature
  2. Use step patterns from catalog
  3. Add @needs(login-owner) tag if login is required
  4. Run /generate-e2e my-feature
  5. Run /run-e2e my-feature --headed
  6. Check results, adjust .feature if needed

Add a new test scenario (natural language)

  1. Run /describe-e2e my-feature
  2. Describe what you want to test in Vietnamese or English
  3. AI generates e2e/features/my-feature.feature
  4. Run /generate-e2e my-feature
  5. Run /run-e2e my-feature --headed

Add a new page to config

  1. Add entry to e2e.config.json → pages
  2. Use Given I am on the "my-page" page in feature

Add a new test account

  1. Add entry to e2e.config.json → accounts
  2. Use Given I login as "my-account" with 2FA in feature

Add a reusable suite

  1. Edit config/suites.config.json → add new suite with steps or nested needs
  2. Use @needs(my-suite) tag in your .feature file
  3. Steps are auto-injected into test.beforeEach() on generation

Test needs a step not in catalog

  1. Write the step naturally in your .feature
  2. Generator marks it as // TODO: [AI]
  3. Ask AI to implement the missing step or add to e2e/steps/
